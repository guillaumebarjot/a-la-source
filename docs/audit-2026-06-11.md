### Procedure de comparaison a executer au point commun

Pour mesurer la parite, il faut, depuis le serveur de prod (acces SSH a obtenir
de l'admin YunoHost ou de Guillaume) :

1. **Version de code deployee**
   ```bash
   cd /chemin/app && git rev-parse HEAD && git status
   git log -1 --format='%ci %h %s'
   cat package.json | grep version
   ```
   Comparer le hash avec `origin/main` (97df9de au 07/06). Si le code de prod
   n'est pas un clone git (copie YunoHost figee), comparer les fichiers
   (`server/src/db/schema.sql`, `package.json`) directement.

2. **Schema de la base de prod**
   ```bash
   sqlite3 /chemin/a-la-source.db '.schema' > /tmp/prod-schema.sql
   sqlite3 /chemin/a-la-source.db \
     "SELECT name FROM sqlite_master WHERE type='table' ORDER BY name;"
   ```
   Comparer la liste des tables et colonnes avec le schema v3 du depot. NB :
   `auto-migrate.ts` rattrape les ecarts additifs au boot, donc une base de prod
   plus ancienne se met a niveau au redemarrage (migrations idempotentes).

3. **Donnees**
   ```bash
   sqlite3 /chemin/a-la-source.db \
     "SELECT 'sources',COUNT(*) FROM sources
      UNION ALL SELECT 'utilisateurs',COUNT(*) FROM utilisateurs;"
   sqlite3 /chemin/a-la-source.db "SELECT id,nom,role,actif FROM utilisateurs;"
   ```
   La base de prod a sa propre population (utilisateurs SSO reels). Elle ne doit
   PAS etre ecrasee par la base de dev OneDrive. Une migration de donnees
   eventuelle se discute separement.

## 2. Documentation

### Etat constate (deja solide)

Le depot est **bien documente** pour un projet associatif :
`README.md` (riche), `docs/architecture.md`, `docs/CHANGELOG.md` (detaille),
`docs/deploiement.md`, `docs/utilisation.md`, `docs/guide-facilitateur.md`,
`docs/mecanismes.md`, `CONTRIBUTING.md`, captures d'ecran.

### Defauts releves (corriges ou a corriger)

- **Incoherence de nommage de role.** Le code et la base utilisent le role
  `animateur` (CHECK `role IN ('membre','animateur','admin')` dans
  `schema.sql`, et `requireRole('animateur','admin')`). La documentation parle
  parfois de `facilitateur` / `facilitateur·ice` (`CONTRIBUTING.md`,
  `README.md`). C'est le **meme** role, mais le mot differe entre code et doc.
  -> A trancher : garder `animateur` (valeur technique) et expliquer que
  l'etiquette affichee est « facilitateur·ice ». Documente dans
  `docs/acces-identite.md`.

- **Bug cosmetique dans `schema.sql`** (non bloquant car la base reelle est
  generee par SQLite, pas par ce fichier, et `auto-migrate.ts` fait foi au
  runtime). Lignes 182-185 : tables internes FTS5 declarees avec un double
  `IF NOT EXISTS` errone :
  ```sql
  CREATE TABLE IF NOT EXISTS IF NOT EXISTS 'sources_fts_data'(...)
  ```
  Ces tables `sources_fts_*` sont des tables shadow gerees automatiquement par
  le module FTS5 a partir de la `CREATE VIRTUAL TABLE sources_fts` : elles ne
  doivent pas etre recreees a la main. Ce sont des artefacts d'un dump. A
  nettoyer dans une passe ulterieure (ne change rien au comportement, mais le
  `schema.sql` echouerait si on tentait de le rejouer tel quel sur une base
  vide). **Non corrige dans cet audit** pour ne rien casser sans test ;
  consigne dans le plan.

- **`docs/deploiement.md`** : le service systemd pointe vers
  `server/src/index.js` (source TS non compilee) alors que le build produit
  `server/dist/index.js`. A corriger pour coller au build reel (`npm run build`
  produit `server/dist/`). Consigne dans le plan.

- **Schema BDD non documente de maniere lisible.** Le seul « schema » est le
  `schema.sql` (artefact de dump, peu pedagogique). Ajout d'un
  `docs/schema-bdd.md` decrivant les grands ensembles de tables (socle, sujets,
  activites-pipelines, observatoire, utilisateurs/lectures).

### Ameliorations faites dans cet audit

- `docs/schema-bdd.md` : carte lisible du schema (par familles de tables).
- `docs/workflow-git.md` : workflow de branches / MR / versions (cf. section 3).
- `docs/acces-identite.md` : modele d'authentification, roles, identite (cf. 4).
- `docs/audit-2026-06-11.md` : le present rapport.

## 4. Droits d'acces, utilisateurs, identite

### Etat constate (mecanisme reel)

- **Authentification = SSO YunoHost**, via le header HTTP `Remote-User`
  (`server/src/lib/auth.ts`, `authMiddleware`). Aucun mot de passe gere par
  l'appli, aucune table de credentials.
- **Auto-provisioning** : au premier passage d'un utilisateur SSO inconnu, un
  compte local est cree avec le role par defaut `membre`
  (`INSERT OR IGNORE INTO utilisateurs`).
- **Fallback dev** : hors production, l'utilisateur est `HydroLooney` par defaut,
  surchargeables par `?_user=Nom`. (Attention : ce fallback ne doit JAMAIS
  s'activer en prod ; il est garde par `NODE_ENV !== 'production'`.)
- **Trois roles** : `membre`, `animateur`, `admin` (CHECK en base). Garde
  cote serveur via `requireRole(...)`. L'admin gere les roles via
  `PATCH /api/auth/users/:id`.
- **Population reelle (base OneDrive)** : 3 comptes. `HydroLooney`=admin,
  `JonLuk`=membre, `Aurelie`=membre. **Aucun compte `animateur`** : les routes
  reservees `animateur`/`admin` ne sont donc aujourd'hui accessibles qu'a
  l'admin.
- **Routes publiques voulues** : `/partage/*` (pages OpenGraph) doivent etre
  declarees publiques dans le SSO YunoHost (`skipped_uris`), sinon le SSO les
  intercepte.

### Risques et points d'attention

1. **La base de travail vit sur OneDrive ARTELIA (compte professionnel).**
   Donnees d'une association militante stockees sur l'espace cloud d'un
   employeur. C'est un risque de confidentialite/gouvernance a part entiere
   (acces ARTELIA potentiel, melange perso/pro, dependance a un compte tiers).
   La prod, elle, doit utiliser une base **locale au serveur YunoHost**, pas
   OneDrive. A clarifier : OneDrive = poste de dev de Guillaume uniquement.

2. **Confiance totale dans le header `Remote-User`.** Si un jour Node etait
   joignable directement (hors nginx YunoHost), n'importe qui pourrait
   usurper une identite via un header `Remote-User` force. En prod, Node DOIT
   n'ecouter que sur `127.0.0.1` et n'etre atteignable que derriere le proxy
   SSO. A verifier au point commun (binding et pare-feu).

3. **Incoherence de vocabulaire** `animateur` vs `facilitateur` (cf. section 2).

4. **Pas de tracabilite d'attribution de role** : un changement de role n'est
   pas journalise. Acceptable a 3 utilisateurs ; a noter si la base grandit.

### Modele d'acces recommande

Deux options coherentes avec l'ecosysteme PIAF :

- **Option A - rester sur le SSO YunoHost natif** (statu quo ameliore). Simple,
  zero dependance, deja en place. On garde `Remote-User`, on durcit le binding
  Node (127.0.0.1), on declare `/partage/` public, on documente la matrice
  role/permission. Recommandee tant que A la source reste autonome sur son
  YunoHost.

- **Option B - adosser a Authentik** (comme le reste de PIAF), si l'on veut une
  identite unifiee entre les outils PIAF. YunoHost peut deleguer le SSO a un
  fournisseur OIDC externe ; l'appli continuerait de recevoir l'identite via le
  proxy (ou via OIDC direct, ce qui demanderait du code). Plus de cohesion
  d'identite, mais plus de couplage et de travail.

Recommandation : **Option A pour l'immediat** (rien a recoder, juste durcir +
documenter), et **garder l'Option B comme cible** si A la source rejoint le SSO
commun PIAF. La decision se prend au point commun.
